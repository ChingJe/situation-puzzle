# 前端設計

## 技術選型

- React
- Vite
- TypeScript

第一版不加入前端測試框架，只要求 TypeScript 與 production build 通過。

## 版面結構

使用單頁 App，主要以 Tab 切換：

- `目前遊戲`
- `歷史紀錄`

## 目前遊戲 Tab

主要區塊：

- 新遊戲主題輸入
- 目前謎面
- 問答紀錄
- 提問輸入
- 提交解答
- 放棄並查看答案

## 歷史紀錄 Tab

主要區塊：

- 已結束遊戲列表
- 歷史遊戲詳情
- 對話紀錄形式的問答渲染
- 玩家提交解答紀錄
- 完整真相
- 結束狀態：已解開或已放棄

## 互動規則

### 建立新遊戲

玩家輸入主題後按下建立遊戲。

若目前已有 `playing` 狀態遊戲，前端應跳確認：

```text
目前遊戲尚未結束，是否開始新遊戲？
```

確認後才建立新局。舊局可以留在後端記憶體中，但前端切換到新局。

### 前端刷新恢復

前端保存 `current_game_id`，例如使用 `localStorage`。

刷新後：

1. 讀取 `current_game_id`。
2. 呼叫 `GET /api/games/{game_id}`。
3. 若成功，恢復目前遊戲。
4. 若失敗，清除 `current_game_id` 並回到初始狀態。

後端重啟導致進行中遊戲失效，第一版不處理。

### 提問

玩家送出問題後：

- 清空輸入框。
- 禁用送出按鈕直到 API 回應。
- 有效問題加入問答紀錄。
- 無效問題顯示錯誤「請輸入可用是／否回答的問題」，不加入正式問答紀錄。

主持人回答只顯示：

- 是
- 否
- 無關

### 提交解答

玩家按提交解答後：

- API 回傳未解開時，固定顯示「尚未解開」。
- 不顯示提示。
- 不顯示缺少的 key facts。
- 成功解開時顯示「成功解開」與完整真相。

### 放棄

玩家按放棄後：

- 呼叫 `POST /api/games/{game_id}/abandon`。
- 顯示完整真相。
- 遊戲狀態變為 `abandoned`。
- 目前遊戲不可再提問或提交解答。

## UI 狀態

前端至少需要處理：

- 初始狀態：尚未建立遊戲。
- 建立中：等待 AI 生成題目。
- 遊戲中：可以提問與提交解答。
- 提問中：禁用提問按鈕。
- 解答判定中：禁用提交按鈕。
- 已解開：顯示真相。
- 已放棄：顯示真相。
- API 錯誤：顯示錯誤訊息。
- Ollama degraded：可顯示健康狀態提示，但不阻止進入 UI。

## 歷史紀錄渲染

歷史遊戲以對話形式呈現：

```text
玩家：男子死亡了嗎？
主持人：是

玩家：他是被殺的嗎？
主持人：否
```

提交解答紀錄可獨立顯示：

```text
解答嘗試 1：尚未解開
解答嘗試 2：成功解開
```
