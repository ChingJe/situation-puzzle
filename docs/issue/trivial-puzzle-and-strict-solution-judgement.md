## 本局內容

玩家看到的謎面：

```text
深夜打烊後，顧客路過便利商店門外，看見店員不斷從紙箱拿出商品往貨架上排列。顧客以為店員正將商品偷偷塞進口袋，懷疑對方趁夜私自取用店裡財物。
```

隱藏真相摘要：

```text
店員打烊後繼續從補貨箱拿出商品並擺到貨架上。顧客不熟悉打烊後補貨流程，將正常補貨誤認為店員偷竊。
```

key facts：

- 店員於營業時間結束後繼續整理貨架商品
- 顧客觀察到店員從補貨箱取出商品擺放至貨架
- 顧客誤解該動作為私自取用商品
- 店員實際行為屬常規補貨作業無違規
- 顧客誤判源於對打烊後工作流程不熟悉

玩家問答：

- Q: 店員實際上有竊取行為嗎

玩家提交解答：

```text
店員就只是在正常補貨
```

`judge_solution` 判定：

```json
{"solved": false}
```

## 觀察問題

### 1. 題目過於直覺，缺少海龜湯反轉

本局的核心真相是「店員只是在正常補貨」。這是便利商店場景中最直覺、最常見的解釋之一，玩家只要問一句「是不是正常補貨」就能接近答案。

這種題目雖然符合日常誤會、視角誤判與單一因果鏈，但不具備足夠的可玩性。它缺少：

- 具體且有辨識度的關鍵物件。
- 非直覺但合理的誤導機制。
- 玩家需要逐步確認的隱藏條件。
- 解答時能形成「原來如此」的轉折。

### 2. Reviewer 只檢查一致性，沒有檢查可玩性

本局 reviewer 通過，原因是謎面、truth 與 key facts 彼此一致：

- 謎面中的客觀描述沒有被 truth 否定。
- surface story 的異常可由 truth 解釋。
- key facts 覆蓋了 truth 中的主要因果。

但 reviewer 沒有擋下「答案太普通、太直覺、太容易被一句話概括」的題目。這代表目前 reviewer contract 偏向一致性檢查，不足以判斷遊戲品質。

### 3. Judge solution 對簡單題過嚴

玩家提交「店員就只是在正常補貨」時，模型判定 `solved=false`。reasoning 顯示原因是玩家沒有說出：

- 顧客看到店員從補貨箱拿商品。
- 顧客誤以為店員偷竊。
- 顧客誤判源於不熟悉打烊後流程。

就嚴格 key facts 來看，這個判定符合目前 prompt 規則；但就本局題目品質來看，玩家已經命中主要真相，而且前面的問答已經確認：

- 店員沒有竊取行為。
- 店員確實在正常補貨。

因此玩家感受會是「已經猜到答案，但系統仍說尚未解開」。這會降低遊戲回饋的可信度。

### 4. 生成、reviewer、judge 三者標準不一致

目前三個環節的隱含標準不一致：

- `generate_core_truth` 允許產生極普通的日常流程誤會。
- `review_puzzle` 只要題目一致就放行，不要求足夠反轉或推理深度。
- `judge_solution` 要求玩家完整覆蓋所有 key facts，即使題目本身只是一句常識解釋。

結果是低難度題被接受，但解答判定仍使用較嚴格的完整因果鏈門檻。

## 影響

- 玩家容易覺得題目「沒有謎」。
- 玩家可能透過少量問答就猜中核心，但提交後仍被判未解開。
- `key_facts` 變成過度細碎的判定門檻，而不是自然玩家解答應涵蓋的核心。
- reviewer 通過不代表題目真的可玩。

## 初步改善方向

### 1. 強化 core truth 可玩性要求

`generate_core_truth` 應避免產生以下低可玩性核心：

- 「店員只是在正常補貨」
- 「清潔人員只是在正常打掃」
- 「工作人員只是在例行檢查」
- 「顧客只是看錯」
- 「只是正常流程被誤會」

不是完全禁止日常流程，而是要求日常流程必須包含具體、非直覺但合理的關鍵物件或條件。

### 2. Reviewer 增加 trivial puzzle gate

`review_puzzle` 應檢查：

- 玩家是否能用一句直覺常識解答解開。
- 真相是否只是「正常工作流程」。
- surface story 是否直接把補貨、整理、打掃、盤點等答案線索寫得太明。
- 題目是否缺少具體關鍵物件或隱藏條件。

若不通過，應回到 `generate_core_truth`，而不是只重寫謎面。

### 3. Judge solution 納入既有問答補完

`judge_solution` 應調整為：

- 玩家提交解答可以和既有問答共同判斷。
- 若玩家答案較短，但已有問答已確認缺少的核心事實，可判定 solved=true。
- 對低複雜度題目，不應要求玩家逐字覆蓋所有 key facts。

### 4. Key facts 分層

可考慮把 key facts 分成：

- `required_solution_facts`：提交解答必須命中的核心。
- `supporting_facts`：問答過程可用，但不一定要求提交時完整重述。

本局中「店員實際行為屬常規補貨作業無違規」應是 required fact；「顧客不熟悉打烊後工作流程」可能是 supporting fact，除非題目設計刻意把它作為核心反轉。
